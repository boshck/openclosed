# AGENTS

Назначение: универсальная инструкция для проектов, где разрабатываются Telegram-боты на Go.

`AGENTS.md` хранит только правила, которые можно перенести в другой проект без изменения смысла. Факты конкретного проекта, карту модулей, команды, окружение, runtime policy, known issues и решения владельца нужно фиксировать в `docs/PROJECT.md`.

## 1) Базовый профиль проектов

- Основной язык разработки: Go.
- Основной тип продукта: Telegram-боты.
- В корне каждого проекта должен быть как минимум один reference symlink с именем `refPROJECT`.
- В каждом проекте должен быть файл `docs/PROJECT.md`.
- Если `docs/PROJECT.md` отсутствует, исполнитель создает его до начала содержательных проектных правок.
- `docs/PROJECT.md` является проектным навигатором: где смотреть код, документы, команды, runtime, env, generated-файлы, ref-ссылки и локальные решения владельца.
- `AGENTS.md` не заполняется локальными фактами проекта. Для этого используется только `docs/PROJECT.md`.

## 2) Приоритет источников

При работе над задачей источники применяются в таком порядке:

1. Явное решение владельца в текущем диалоге.
2. `AGENTS.md`.
3. `docs/PROJECT.md`.
4. Актуальные документы проекта, перечисленные в `docs/PROJECT.md`.
5. Код, логи и фактическое поведение системы.

Если документы и код расходятся:

- зафиксировать расхождение;
- считать фактическое поведение по коду и логам подтвержденным фактом;
- согласовать целевое поведение с владельцем;
- после согласования привести код и документы к консистентности.

## 3) Общая дисциплина

- Сначала понять цель, границы и ожидаемый результат.
- Если запрос двусмысленный и безопасное допущение невозможно, задать один короткий вопрос.
- Держать правки строго в границах задачи.
- Не откатывать чужие изменения без прямой просьбы владельца.
- Не выдавать гипотезы за факты.
- Не выдумывать требования, архитектуру, API, ограничения, команды или поведение системы.
- Давать только проверенную информацию. Неподтвержденное явно помечать как `гипотеза` или `открытый вопрос`.
- Неподтвержденную логику фиксировать только как `draft`.
- В рабочих документах разделять `Подтверждено`, `Гипотезы` и `Открытые вопросы`.
- Не заполнять `TBD` догадками.
- Перед мини-правками давать мини-план. Перед крупными правками давать большой план.
- По ходу работы давать короткие статусы прогресса.
- После изменений кратко сообщать: что изменено, где, зачем, как проверено.
- Если решение спорное или сложное, сначала согласовать UX-результат, потом выбирать техническую реализацию.
- Для диагностики сначала собрать факты: шаги воспроизведения, симптомы, логи, места в коде.
- Если фактов недостаточно, явно назвать недостающие данные и запросить их.
- Для инцидентов сначала дать короткий прямой ответ, затем подтвержденные факты, затем план проверки или правки.
- Не повторять уже принятый владельцем вывод без новой информации.
- Если владелец просит короче, оставить только ответ, нужные факты и следующий шаг.
- Смежные риски выносить отдельно в `Дополнительно`, не подменяя ответ на исходный вопрос.

## 4) HARD-STOP перед правками

Любые изменения кода, документов, данных, окружения, миграций, генерации и форматирования выполняются только после явного подтверждения владельца.

До подтверждения разрешены только:

- вопросы;
- анализ;
- чтение файлов;
- подготовка вариантов;
- подготовка плана;
- перезапись `comm`, если в проекте принят owner-run workflow из раздела 10.

До подтверждения запрещено:

- изменять файлы, кроме разрешенного `comm`;
- запускать команды с записью в файлы, БД или окружение;
- запускать генерацию, форматирование, миграции и массовые правки данных.

Перед стартом правок обязателен `Чек перед правками`:

- цель;
- границы;
- DoD;
- список файлов;
- проверки после правок.

Если HARD-STOP нарушен:

- ответ считается невалидным;
- первое действие исполнителя - откатить свои правки по текущей задаче;
- после отката исполнитель возвращается в режим анализа и ждет подтверждение владельца.

## 5) Протокол ответа до правок

Для новой задачи или вопроса исполнитель сначала дает блоки в таком порядке:

1. `Как понял задачу`: что нужно изменить, где будет результат, что не входит в границы.
2. `Подтвержденные факты`: только факты из кода, документов, логов или слов владельца.
3. `Гипотезы`: если есть предположения и какие данные нужны для проверки.
4. `Dependency Audit`.
5. `Предлагаемое решение`: default-вариант, альтернативы, UX/UI-эффект или `UI не меняется`.
6. `Что делать владельцу сейчас`: если нужно подтверждение, действие или артефакт.
7. `Чек перед правками`: если задача требует изменений.

Запрещено смешивать подтвержденные факты и гипотезы в одном списке.

Для многоэтапных задач нужны `Gate-критерии`: что выполнено, как проверяется, кто подтверждает переход. Следующий этап не стартует до прохождения gate.

## 6) UX-first

- Для продуктовых и пользовательских вопросов базовый язык обсуждения - UX-сценарии и ожидаемое поведение пользователя.
- Технические термины допустимы только с переводом на UX-язык.
- Каждый default-вариант должен описывать, где пользователь увидит результат: экран, блок, элемент, состояние, текст, callback-toast или отсутствие UI-изменений.
- Для диагностики сначала закрывается исходный UX-сбой, потом обсуждаются смежные улучшения.
- Для найденной дыры или противоречия нужно предложить конкретное изменение, указать места изменений и дождаться подтверждения до правок.

## 7) Dependency Audit и анти-правка-ради-правки

Перед правками обязателен `Dependency Audit`:

- затрагиваемые UX-сценарии и модули;
- зависимости по данным, лимитам, статусам, навигации и внешним контрактам;
- возможные побочные эффекты и регрессии;
- UX-несостыковки и логические противоречия с приоритетом.

До закрытия критичных несостыковок по текущей задаче к правкам не приступать.

Каждая правка должна иметь заранее заданный измеримый результат, привязанный к цели. Для инцидента заранее фиксируется:

- какой симптом должен уйти;
- на каком окне времени проверяется;
- какой порог считается прохождением.

Перед внедрением пройти `10x Goal-Alignment Check`: десять раз проверить, является ли правка прямым вкладом в цель, а не правкой ради правки. Если есть сомнение, правку перепроектировать.

После этого выполнить `Check of Check`: отдельно проверить, не пропущена ли в самой проверке правка ради правки.

## 8) Runtime policy

Runtime-режим проекта фиксируется только в `docs/PROJECT.md` после решения владельца или фактической проверки.

Пока runtime policy не зафиксирована:

- не выполнять локальные runtime-команды, влияющие на данные или окружение;
- для server-only проектов готовить paste-ready команды или файлы для владельца;
- не считать локальные проверки доказательством состояния рабочей системы.

В `docs/PROJECT.md` нужно фиксировать:

- где существует рабочий runtime: local, staging, prod, server-only;
- какие runtime-команды разрешены исполнителю;
- какие команды запрещены без отдельного подтверждения;
- какие команды являются owner-run only.

## 9) Live/API-проверки

Если live-запрос, внешний API, интеграционный запрос или диагностическая команда не сработали:

1. После первой неудачи остановиться и собрать факты: HTTP status, error code/message, endpoint, method, headers без секретов, body schema без секретов, отличие от успешного примера.
2. Перед второй попыткой проверить документацию, локальные материалы, официальные источники или успешный пример владельца.
3. Не менять несколько неизвестных параметров за одну попытку без причины.
4. Если вторая попытка не сработала, запросить у владельца недостающие данные: curl/fetch, headers без секретов, endpoint, method, body, query, account/token scope, screenshot или Network export.
5. Третью попытку выполнять только после новой информации или точечной гипотезы.
6. Если третья попытка не сработала, отдать владельцу факты и список недостающих данных.

Четвертая попытка запрещена без явного текстового разрешения владельца.

## 10) Серверная диагностика и `comm`

Раздел применяется только если в `docs/PROJECT.md` зафиксирован owner-run workflow для серверной диагностики.

- Рабочая диагностика выполняется на сервере.
- Локальные запросы к БД и локальное состояние не считаются валидной диагностикой рабочей системы, если владелец не решил иначе.
- Исполнитель подготавливает и перезаписывает файл `comm` в корне репозитория.
- Перезапись `comm` для серверной диагностики разрешена без отдельного подтверждения.
- `comm` не является исполняемым скриптом по умолчанию; это поле с командами для ручного copy/paste в SSH.
- В `comm` писать только готовые команды: одна команда на строку, без heredoc, функций, bash-скриптов и ручных плейсхолдеров.
- Разрешены диагностические команды: логи, read-only SQL, статусы сервисов, контейнеров, образов и процессов.
- Разрешены точечные SQL-фиксы только по подтвержденной проблеме данных и только для ручного запуска владельцем.
- Запрещены deploy/release/infrastructure команды: pull, up, down, restart, build, миграции, генерация, массовые правки данных.
- Если нужен деплой, перезапуск или изменение окружения, сначала запросить подтверждение в чате.
- Для логов использовать компактные выборки: агрегаты, счетчики, `uniq -c`, короткий `tail`, фильтры по модулю, сущности или сигнатуре.
- Для блочной вставки ставить маркеры `echo STEP_*` перед логическими блоками.
- Выводы по серверной диагностике делать только по результатам, полученным с сервера.

## 11) Paste-ready файлы и команды

Если владелец просит записать команду или код в файл для копирования в терминал или браузерную консоль:

- содержимое должно быть готово к запуску без ручных изменений;
- не добавлять Markdown, пояснения, заголовки, списки шагов или плейсхолдеры, если владелец не просил;
- не использовать `prompt`, ручную вставку токена или ручную подстановку параметров, если владелец не просил интерактивный ввод;
- инструкции давать отдельным сообщением после записи файла.

## 12) Observability

Если проблема не закрыта с первого прохода, причина не локализована или текущие логи не отвечают, где именно ломается сценарий, это дефицит observability.

Default-следующий шаг: предложить и после подтверждения владельца внедрить подробное логирование или трассировку ровно под проблемный контур.

Цель логирования: по одному сценарию видеть цепочку `ingress -> handler/service -> internal calls -> external calls -> result`.

Минимум для каждого наблюдаемого шага:

- start event;
- finish event;
- `elapsed_ms`;
- `result/error`;
- `correlation_id` или `request_id`;
- идентификаторы сущностей, достаточные для связывания цепочки.

Ограничения:

- не логировать секреты, токены, пароли и приватные ключи;
- не логировать чувствительные данные в сыром виде, если достаточно surrogate-полей;
- усиление логирования не должно подменять факты гипотезами.

Если проблема не решена после первой правки, новая итерация диагностики должна ответить:

- каких логов не хватило;
- где их нужно добавить;
- какой новый сигнал должен снять неопределенность.

## 13) Крупные изменения

Крупным считается изменение, если выполняется хотя бы одно условие:

- затронуто более 5 файлов;
- затронуто несколько независимых модулей;
- меняется более 3 UX-сценариев;
- меняется ключевой пользовательский сценарий;
- меняется публичный API, схема данных или контракт интеграции;
- требуется миграция данных;
- требуется новый инфраструктурный шаг;
- затронуто 2+ модуля с новым поведением.

Для крупного изменения обязательно:

- сначала отдельный рабочий документ с решением;
- затем реализация;
- затем сверка `задокументировано` vs `фактически работает`.

Документ крупного изменения должен содержать:

- проблема и текущие симптомы;
- `as-is`;
- `target behavior`;
- границы: что делаем и что не делаем;
- план реализации;
- почему выбран этот вариант;
- риски и план отката;
- DoD и gate-критерии;
- goal metrics, окно наблюдения и pass/fail gate для архитектурного rework или инцидента.

## 14) Инциденты и change docs

- Для одного активного инцидента допускается только один основной `incident doc`.
- Все новые server facts, gate-решения и verdict по инциденту фиксируются только в основном `incident doc`.
- Остальные документы могут ссылаться на него, но не подменять его.
- Если расследование причины и проект целевых изменений идут одновременно, нужны два документа: `incident doc` и `change doc`.
- `incident doc` хранит симптомы, подтвержденные server/code facts, гипотезы, gate-критерии и журнал расследования.
- `change doc` хранит target-state, инварианты, границы решения, migration plan, риски и goal metrics.
- Если новый факт опроверг старые выводы, в той же итерации обновить основной документ и пометить устаревшие формулировки как `historical snapshot`, `stale` или `history`.
- Нельзя оставлять несколько актуальных документов с противоречащими текущими выводами.
- Формулировки о внешнем API, runtime, интеграции или инфраструктуре должны опираться на первоисточники с отделением факта от вывода по проекту.
- После каждого нового лога, дампа или server fact обновлять журнал расследования: `done`, `pending`, `next`.

## 15) Project docs

`docs/PROJECT.md` должен содержать только проверенные проектные факты или явно помеченные гипотезы.

Минимальный состав:

- назначение проекта;
- ref-ссылки и назначение `refPROJECT`;
- карта модулей;
- актуальные документы;
- команды разработки и проверки;
- runtime policy;
- конфигурация и секреты;
- generated/managed files;
- что не редактировать вручную;
- открытые вопросы.

Статусы документов:

- `actual` - актуальный источник для работы;
- `draft` - черновик, требует проверки или дополнения;
- `stale` - устаревший документ, не источник истины;
- `history` - исторический контекст.

Готовность документа определяется не статусом, а соответствием текущему коду, логам и решениям владельца.
