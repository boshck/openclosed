# Guard Mode

Статус: actual

## Проблема

Владелец хочет публичные Telegram-каналы, группы и супергруппы, которые можно открыть и читать, но где обычный пользователь не должен оставаться подписчиком или участником. Если не-админ пишет сообщение, оно должно быть удалено.

## As-is

Подтверждено:

- Прикладной код на старте отсутствовал.
- `docs/PROJECT.md` отсутствовал и создан перед содержательными проектными правками.

## Target behavior

Подтверждено:

- Бот получает `chat_member` updates.
- Если пользователь стал `member` или активным `restricted` участником и не является админом, владельцем, самим ботом или allowlist-пользователем, бот ставит его в persistent queue на удаление.
- Удаление выполняется через `unbanChatMember` без `only_if_banned`, чтобы убрать пользователя из чата без постоянного бана.
- Если не-админ пишет сообщение в группе или супергруппе, бот удаляет сообщение через `deleteMessage` и ставит автора в очередь удаления.
- После рестарта бот продолжает незавершенную очередь удаления.
- Все действия пишутся в журнал модерации без токенов и секретов.

## Границы

Делаем:

- Long-polling бот на Go.
- PostgreSQL-хранилище для известных участников, очереди удаления, allowlist и журнала событий.
- Обработчики `chat_member` и `message`.
- Unit-тесты ключевых guard-решений.

Не делаем:

- Админ-панель.
- Платные подписки.
- Скоринг фейков и антинакрутку.
- Live/API-проверку реального Telegram-бота.
- Деплой и systemd/docker runtime.

## Почему выбран этот вариант

- Telegram Bot API не дает полного списка всех участников канала или группы, поэтому надежная стратегия строится вокруг полученных updates, сообщений и persistent retry-очереди.
- PostgreSQL выбран по решению владельца от 2026-05-08.
- `unbanChatMember` соответствует UX: пользователь удаляется, но не получает постоянный запрет на повторное открытие публичного чата или канала.

## Риски

- Если бот не работал дольше срока хранения updates Telegram, он не узнает о части вступивших пользователей через чистый Bot API.
- Пользователь может быть участником короткое время между вступлением и обработкой update.
- Если Telegram API временно недоступен или у бота нет нужных прав, удаление будет ретраиться, но пользователь останется до успешной попытки.
- Для удаления сообщений нужны права администратора на удаление сообщений.

## DoD

- `go test ./...` проходит.
- `go test -race ./...` проходит или причина невозможности зафиксирована.
- Вступление не-админа приводит к записи в очередь удаления.
- Сообщение не-админа в группе или супергруппе удаляется и автор ставится в очередь удаления.
- Сообщение админа не удаляется.
- Очередь удаления сохраняется в PostgreSQL и обрабатывается после рестарта.

## Gate-критерии

- Gate 1: код компилируется и unit-тесты проходят локально.
- Gate 2: владелец запускает бота в тестовом канале или группе с нужными правами.
- Gate 3: в тесте пользователь вступает, удаляется без постоянного бана и может снова открыть публичную ссылку.
- Gate 4: сообщение не-админа удаляется, сообщение админа остается.
